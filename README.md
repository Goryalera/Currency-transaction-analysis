# Currency Transaction Analysis

Учебный аналитический проект по теме:

**Оптимизация front-to-back процесса внебиржевой валютной сделки корпоративного клиента**


Проект моделирует полный путь OTC FX Spot сделки: запрос клиента, проверка данных и лимитов, котировка, согласование, регистрация сделки, подтверждение, расчеты, сверка и обработка ошибок.

Важно:
проект не имитирует внутренние процессы конкретного банка. Операционный журнал сделок является синтетическим, а рыночная часть данных построена на публичном Kaggle dataset.

## Цель проекта

Проанализировать AS-IS процесс проведения внебиржевой валютной сделки корпоративного клиента, найти узкие места, описать требования и спроектировать TO-BE процесс с частичной автоматизацией.

Проект показывает навыки:

- понимание FX Spot продукта и жизненного цикла сделки;
- описание end-to-end банковского процесса;
- анализ AS-IS и проектирование TO-BE;
- BPMN-моделирование;
- расчет процессных метрик через Python и SQL;
- выявление bottlenecks и операционных рисков;
- подготовка бизнес-, функциональных и нефункциональных требований;
- оценка экономического эффекта автоматизации;
- оформление результатов в dashboard, Excel и PDF.

## Данные

Использован гибридный подход:

- **Kaggle market data**: ежедневные FX-курсы из dataset `asaniczka/forex-exchange-rate-since-2004-updated-daily`;
- **Synthetic process data**: сгенерированные процессные события и статусы сделки.

Публичные Kaggle-датасеты хорошо подходят для рыночных курсов, но обычно не содержат внутренний банковский журнал обработки сделок. Поэтому поля вроде `request_time`, `quote_time`, `limit_check_time`, `trade_capture_time`, `confirmation_time`, `settlement_time`, `manual_processing_flag`, `rework_count`, `exception_type` и `sla_breach` сгенерированы синтетически по заданным бизнес-правилам.

## Реализованный сценарий

Корпоративный клиент хочет купить или продать валюту через банк. В AS-IS процессе часть операций выполняется вручную: сотрудники переносят данные между системами, проводят проверки, согласовывают условия, регистрируют сделку и обрабатывают ошибки.

Основные проблемы процесса:

- долгое проведение сделки;
- повторный ручной ввод данных;
- риск ошибки в сумме, валюте или расчетных реквизитах;
- поздняя проверка лимитов;
- просрочка подтверждения;
- проблемы при расчетах;
- отсутствие единого мониторинга SLA и исключений.

## Основные выводы

Крупнейшие зоны потерь в AS-IS процессе связаны не с рыночной котировкой, а с ожиданием, ручными передачами данных и поздними проверками.

Подтвержденные bottlenecks:

- задержки на этапе расчетов и сверки;
- ручная регистрация сделки после согласования условий;
- поздняя проверка лимитов;
- неполные settlement instructions;
- отсутствие отдельной очереди исключений;
- реактивный, а не превентивный контроль SLA.

## TO-BE идея

Целевой процесс строится вокруг принципа:

**Стандартные сделки проходят автоматически, сотрудники работают преимущественно с исключениями.**

Ключевые изменения TO-BE:

- единая форма запроса;
- автоматическая проверка обязательных полей;
- pre-trade проверка лимитов до передачи трейдеру;
- единый `deal_id` на всем жизненном цикле;
- автоматическая передача данных в trade capture;
- автоматическое формирование подтверждения;
- отдельная очередь исключений;
- SLA timer и уведомления;
- аудит всех изменений сделки.

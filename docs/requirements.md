# Requirements

## Business Requirements

| ID | Requirement |
|---|---|
| BR-001 | Сократить среднее время проведения стандартной сделки |
| BR-002 | Снизить количество ручных операций |
| BR-003 | Снизить долю сделок, возвращаемых на исправление |
| BR-004 | Обеспечить контроль соблюдения SLA |

## Functional Requirements

| ID | Requirement | Priority | AS-IS problem |
|---|---|---|---|
| FR-001 | Система должна автоматически проверять заполнение обязательных полей | Must | Неполные реквизиты выявляются поздно |
| FR-002 | Система должна выполнять проверку доступного лимита до передачи запроса трейдеру | Must | Лимит проверяется после затрат на котировку |
| FR-003 | Каждой сделке должен присваиваться единый идентификатор | Must | Есть риск дублей |
| FR-004 | Данные подтвержденной сделки должны автоматически передаваться в систему учета | Should | Повторный ручной ввод |
| FR-005 | Сделки с ошибками должны направляться в отдельную очередь исключений | Should | Стандартные и проблемные сделки идут одним маршрутом |
| FR-006 | Система должна уведомлять ответственного сотрудника о приближении нарушения SLA | Should | Нет раннего мониторинга SLA |

## Non-Functional Requirements

| ID | Requirement |
|---|---|
| NFR-001 | Все изменения сделки должны сохраняться в журнале аудита |
| NFR-002 | Доступ к изменению сделки должен предоставляться в соответствии с ролью пользователя |
| NFR-003 | Повторная отправка запроса не должна создавать дублирующую сделку |

## Business Rules

| ID | Rule |
|---|---|
| BRULE-001 | Сделка не может быть подтверждена при недостаточном лимите |
| BRULE-002 | Изменение суммы или валютной пары после подтверждения требует повторного согласования |
| BRULE-003 | Сделки выше установленного порога требуют дополнительного контроля |

## User Stories

| ID | User Story | Acceptance Criteria |
|---|---|---|
| US-001 | Как Sales, я хочу видеть ошибки обязательных полей до отправки запроса дальше, чтобы не возвращать сделку на исправление после котировки | Незаполненные обязательные поля подсвечены; сделка не уходит трейдеру до исправления |
| US-002 | Как Risk Officer, я хочу, чтобы лимит проверялся до котировки, чтобы отклоненные сделки не занимали время Trader | При недостаточном лимите создается отказ; Trader не получает задачу |
| US-003 | Как Operations, я хочу видеть очередь исключений, чтобы быстрее обрабатывать settlement fail и incomplete SSI | Все исключения имеют тип, владельца, срок SLA и статус обработки |
